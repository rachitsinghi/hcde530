C6 — Data Visualization
My MP1b notebook (mp1b_visualizations.ipynb) makes three charts, each chosen using the Abela Chart Chooser framework so the chart type matches the question rather than defaulting to a generic bar or scatter. For Q1 ("can a custom constellation shape be built from real stars?") I picked a 2D sky-projection scatter with a glowing line overlay, because the question is a relationship between two positional variables and the chart needs to read as a star map a human would actually recognize. For Q2 ("do a few bright stars anchor each cluster, or is brightness spread evenly?") I picked a horizontal box plot with diamond overlays for stars at mag < 3.0, because it's a distribution question across many groups and the box plot directly exposes the skew, outliers, and the gap between anchors and dim filler stars, a histogram per cluster would have produced 15 stacked plots that nobody can compare. For Q3 ("do clusters host different stellar populations?") I picked a 100% stacked horizontal bar chart with the spectral classes colored to match real stellar colors (deep blue O, white A, yellow G, red M), because it's a composition question about share-of-total across many groups, and the stacked bar lets you scan from cluster to cluster and see compositional differences at a glance. Each chart has a title that states what it shows, axes labeled with units (apparent magnitude with the note "lower = brighter" because the inverted scale is counterintuitive), and a written justification in the preceding markdown cell. The most useful learning from this assignment was that I initially tried to build the cat constellation in 3D Cartesian space using real distances, and the result was unreadable because the matched stars sat between 49 pc and 411 pc from Earth, so the cat-face polygon stretched into a slab along the line of sight and the shape only resurfaced when I collapsed it back to a 2D sky projection. That was a case of the chart type fighting the data: the right answer wasn't a fancier 3D plot, it was the boring 2D projection that matches how humans actually see the sky. I kept the 3D version in the notebook as Chart 1b to show that contrast honestly, rather than hiding the failed experiment.

C7 — Critical Evaluation and Professional Judgment
Three separate moments in this project pushed me to evaluate AI output rather than trust it. First, when I asked Claude to merge three CSVs into a unified star catalog for MP1, the merged file came back looking complete (every row populated) but Claude flagged that the three files shared no common key and that the 64,000 "stars" in the simulation file were anonymous particles whose physical properties had been randomly paired, a hot blue star might end up nearby and a cool red dwarf far away, which is unphysical. I treated that flag as a real concern instead of accepting the polished-looking output, evaluated the recommended alternative (the HYG database), spot-checked it against known stars like Sirius and Betelgeuse to verify its columns matched published values, and made the call to discard the synthetic file entirely. Second, while building Chart 1 in MP1b, Cursor produced a 3D constellation plot at my request but the cat shape was unrecognizable because the matched stars' real distances stretched the outline along the line of sight; I documented that failure in the notebook's Cell 5 ("the 3D version was abandoned because… the cat shape is projection-dependent") and pivoted to a 2D sky chart, keeping the 3D version alongside as Chart 1b so the limitation stays visible. Third, an earlier draft of the "build your own constellation" extension used a Plotly FigureWidget with dragmode='drawline' to let users sketch a constellation interactively. I caught that this depends on a live Jupyter widget bridge that doesn't render on GitHub or in static notebook viewers, and swapped it for an edit-the-dictionary-and-re-run pattern that actually works in every environment a grader might open the notebook in. The pattern I learned across all three: AI tools produce structurally correct output that's substantively wrong (a complete-looking merged CSV with fabricated correlations, a 3D plot that "works" but is unreadable, a widget that runs locally but breaks on submission), and the only defense is verifying against a ground truth or the actual deployment context, published data in the first case, my own eyes on the rendered chart in the second, knowing how GitHub renders notebooks in the third.
